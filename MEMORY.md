# MEMORY — контекст проекта day-runner

Этот файл — слепок контекста для будущих сессий. Если открываешь проект «с нуля», прочитай его и `ARCHITECTURE.md` — этого должно хватить.

## Зачем проект

Личный «пульт рабочего дня» для поиска работы. Открываешь утром — видишь расписание 10:00–18:00 с подсветкой текущего блока, помидорку, чек-листы по блокам. Каждое утро чек-лист и счётчик спринтов начинаются заново. Публичный, чтобы заодно лежал в портфолио.

Прод: **https://konicaru.github.io/day-runner/**
Репо: **https://github.com/konicaRu/day-runner**

## Стек и допущения (важно — это решение проекта)

- Чистый Vanilla HTML + CSS + JS. **Без фреймворка, без сборки, без npm.**
- Расписание — обычный JS-объект в `schedule.js` (не `fetch` JSON, чтобы файл работал и локально по `file://`, и на Pages без CORS).
- Состояние — `localStorage`.
- Шрифты — Google Fonts по CDN (`Manrope`, `JetBrains Mono`).
- Хостинг — GitHub Pages (`main` / root, `.nojekyll` есть).

Если что-то меняем (например, переходим на React/Vite или Python) — это перепиливает половину архитектуры; сначала обсуждаем.

## Принципы работы (Карпатый)

1. Идём строго по майлстоунам, по одному. После каждого — стоп, показ, ждём «ок».
2. Не трогаем то, что уже работает. Никаких рефакторингов вне скоупа шага.
3. Маленькие функции, без преждевременных абстракций. Непонятно — спрашиваем.
4. Коммит + пуш после каждого зелёного шага (см. правило ниже).

## Правило пуша

**Пушим сразу после каждого зелёного шага**, не накапливаем локально. Сайт задеплоен на GitHub Pages, пользователь хочет видеть прогресс на проде сразу. Сообщения коммитов — `"M2: ..."`, `"R3: ..."`, `"feat: ..."`.

## Кеш-бастинг

В `index.html` ссылки на статику идут с `?v=N` (`styles.css?v=8`, `app.js?v=8`, `schedule.js?v=8`). Pages и браузеры агрессивно кешируют — если меняли CSS/JS, **обязательно бампаем версию во всех трёх ссылках** и пушим. Иначе пользователь видит старое.

## История майлстоунов (что сделано)

Базовый план (`plan_day_runner.md`):

- **M0** — каркас, рендер расписания из `schedule.js`, часы, подсветка текущего блока.
- **M1** — чекбоксы сохраняются в `localStorage`, сбрасываются автоматически в новый день. Плюс добавили обратный отсчёт до 18:00 в шапке (это была доп. просьба, не из плана).
- **M2** — Pomodoro-таймер: старт/пауза/пропуск/сброс, авто-переход работа→перерыв→работа, длинный перерыв 15 мин после 4 спринтов, пресет 25/5 ↔ 50/10. Счётчик спринтов и пресет хранятся в `localStorage`, обнуляются в новый день.
- **M3** — сигналы: двойной бип через Web Audio + системные уведомления через Notification API. Тумблер звука и кнопка запроса разрешения. Триггеры: конец спринта, конец перерыва, смена блока расписания. Всё в `try/catch`. Доп. фича сверху плана: **выбор звука сигнала** — 5 встроенных пресетов (бип/колокольчик/динь/тройной/низкий) + загрузка своего файла (хранится в `localStorage` как data URL, лимит ~1 МБ) + кнопка тестового воспроизведения.
- **M5** (выполнен после M0, по просьбе) — деплой на Pages. Pages включён через UI (Settings → Pages → main / root). Файл `.nojekyll` лежит в корне.
- **M6** — счётчик отправленных резюме: пилюля «Резюме N/цель» с кнопками −/+ в шапке (строка `topbar-now`, справа). Цель на день настраивается (по умолчанию 10), при достижении пилюля зеленеет. Клик по пилюле — поповер: мини-график за 14 дней, таблица «дата → количество → ✓», суммы за 7 дней и за всё время. История в `localStorage` (`day-runner-resumes`), **не сбрасывается** в новый день. Это первый шаг к стретчу «трекер откликов».
- **M7** — плавающее окно счётчика через Document Picture-in-Picture (кнопка `⧉` рядом с «+»). Мини-окно поверх всех окон с числом и −/+, синхронизировано со страницей (один JS-контекст). Только Chrome/Edge 116+ (у пользователя Chrome — ок); в остальных браузерах кнопка скрыта. Вкладка day-runner должна оставаться открытой, открытие — по клику (политика user gesture).
- **M9** — графики в `STATS.md` через Mermaid `xychart-beta` (GitHub рендерит сам): столбики за последние 30 дней; когда накопится больше одного месяца — добавляется секция «По месяцам» с графиком и таблицей. Рендер на GitHub проверен (страница blob, SVG отрисовывается).
- **M8** — автозапись `STATS.md` (markdown-таблица «дата → резюме» + итого) в этот же репозиторий через GitHub Contents API. Пользователь сам создаёт fine-grained PAT (Contents: write, только этот репо) и вставляет в поле в поповере; токен живёт в `localStorage`, в репо не попадает. Дебаунс 2 мин после последнего «+», досинк при загрузке, кнопка «Синк» для немедленной записи. Пользователь согласился, что статистика публичная. Токен истекает максимум через год — потом завести новый.

Редизайн (`plan_redesign_day_runner.md`):

- **R1** — дизайн-токены и каркас: CSS-переменные (цвет/радиусы/тени/шрифты), фон `#F6F7F9`, акцент индиго `#6366F1`, шрифты `Manrope` + `JetBrains Mono`.
- **R2** — карточки блоков: цветные теги типа (План/Фокус/Учёба/Отдых), прогресс `N/M` + тонкая полоска, кастомные чекбоксы (заливка акцентом, белая галочка, зачёркивание текста), hover-эффекты.
- **R3** — помидорка-герой: большое круговое SVG-кольцо (радиус 104, обводка 12), цвет меняется по режиму, опустошается по мере убывания времени, drop-shadow при работе.
- **R4** — липкая шапка в две строки с blur-фоном: дата, обратный отсчёт до 18:00, часы, пилюля «сейчас по плану» с цветом по типу блока и человекочитаемым отсчётом до смены (`до смены 1ч 06м`). Адаптив: ≤720px и ≤420px. Стаггер появления карточек на загрузке.

## Решения по дизайну, которые держим

- Один акцент — индиго `#6366F1`. Hover — `#4F46E5`. Светлая заливка — `#EEF0FE`.
- Карточки: радиус 16px, тонкая граница, мягкая тень, лёгкий подъём на hover.
- Семантические цвета тегов по типу блока (см. `--tag-*` переменные в `styles.css`).
- Шрифт текста — `Manrope`, цифры (часы/таймер) — `JetBrains Mono` с `tabular-nums`.
- Кастомные чекбоксы (нативные браузерные скрыты `opacity: 0; position: absolute`).
- Микро-анимации через `--transition` (180ms ease).

## Планируемая фича: сбор статистики откликов с hh.ru (в ОБСУЖДЕНИИ, код не писан)

Состояние на 2026-06-11: идею обсудили и договорились о подходе, но **ни строки кода не написано**. Следующий конкретный шаг — «пробник» (см. ниже). Это развитие стретча «трекер откликов» — hh.ru как источник правды вместо ручной кнопки «+».

**Цель.** Собирать число отправленных откликов по дням прямо с hh.ru (а не вбивать руками). Пользователь уволился 1 мая 2026 — хотим собрать всю историю с этой даты и дальше держать актуальной.

**Страница-источник.** «Отклики и приглашения»: `cherepovets.hh.ru/applicant/negotiations` (поддомен города — Череповец). Это отдельная страница, НЕ просмотр вакансий. Отклики отсортированы новые→старые, разложены по страницам с нумерацией 1,2,3,4,5,6. Фильтр «Все 78» = всего откликов. Бейджи «Отклики 25» / «Все 78» — это НЕ «за сегодня».

**Форматы дат на карточке** (под названием компании): `сегодня` / `вчера` / `9 июня` (число + месяц словом, БЕЗ года). 
- Подводный камень 1: слово «сегодня» в карточке встречается дважды — дата отклика И серый тег «Был онлайн сегодня» (статус работодателя). Целиться надо строго в узел даты.
- Подводный камень 2 (ГЛАВНЫЙ, не проверено): неизвестно, «сегодня/вчера/дата» — это дата ОТПРАВКИ отклика или дата ПОСЛЕДНЕЙ АКТИВНОСТИ (например, сегодня пришёл отказ на вчерашний отклик → карточка может показывать «сегодня»). Если последняя активность — простой подсчёт завысит. Надо проверить на живой странице.

**Решение по инструменту: Tampermonkey-юзерскрипт** (НЕ букмарклет). Причина: пользователь хочет, чтобы собиралось САМО при открытой странице, а не по клику. Юзерскрипт расширение само вклеивает в страницу откликов при каждой её загрузке; отдельная вкладка/окно не открывается.

**Механика двух режимов:**
- *Обычный (каждый заход):* читает только страницу 1 (последние дни), перезаписывает эти дни по факту. Самоисцеление: пропустил день — он ещё виден в списке, подтянется при следующем заходе. Если за день столько откликов, что не влезли на стр.1 — скрипт сам доходит до стр.2.
- *Бэкфилл (один раз):* проходит все страницы 1→…→6 назад до 1 мая. Накапливает данные МЕЖДУ перезагрузками страниц в своём хранилище (скрипт авто-срабатывает на каждой загрузке, в т.ч. на ?page=3), останавливается на 1 мая или когда страницы кончились. Пользователь увидит, как страница один раз сама пролистнётся.

**Куда писать данные.** Переиспользовать существующий GitHub-токен (из M8), коммитить только при изменениях. hh становится источником правды; ручная «+» остаётся как запасной ввод. НАДО решить единый источник правды, чтобы два писателя не спорили за `STATS.md` (вероятно: юзерскрипт пишет data-JSON, day-runner его читает и рисует).

**Напоминание.** Живёт в day-runner (его открывают каждое утро): если статистика «протухла» (N дней не обновлялась) — строка/уведомление «загляни в Отклики», опц. кнопка «Открыть отклики hh». Сам hh-скрипт напомнить не может — он не работает, пока hh не открыт.

**Принципиальное ограничение.** Полностью без участия не выйдет: hh отдаёт отклики только в залогиненной сессии в реальной видимой вкладке, тихий фоновый скрейп браузер запретит. Остаётся одна привычка пользователя — иногда открывать страницу Откликов.

**Открытые технические вопросы — решаются ПРОБНИКОМ на живой странице:**
1. Как устроено перелистывание: `?page=N` в URL / кнопка / бесконечная прокрутка (по виду — обычная постраничная, это удобно).
2. Стабильные селекторы (у hh обычно есть `data-qa`-атрибуты — тогда скрипт устойчив).
3. Семантика даты (отправка vs последняя активность — см. подводный камень 2).

**Следующий шаг (согласован):** написать «пробник» — вставка в консоль браузера на странице Откликов, печатает: сколько карточек, тексты их дат, как устроена пагинация. По отчёту собрать настоящий юзерскрипт. Дизайн-предложение: «автомат предлагает число, пользователь подтверждает одним кликом» — на случай кривой семантики даты.

## Известные мелочи / на что обращать внимание

- Часть Web Audio (бип) не сработает до первого клика пользователя — это политика браузеров. Поэтому пользователю надо нажать любую кнопку (Старт/Звук/тест), чтобы AudioContext разблокировался.
- `localStorage` ограничен ~5 МБ. Кастомный аудиофайл лимитирован 1 МБ (raw), чтобы влезал с запасом.
- `?v=N` в ссылках — единственный механизм инвалидации кеша. Если поменял `styles.css` или `app.js` — бампай номер во всех `?v=` в `index.html` (сейчас `?v=12`).
- Используем `Notification.permission` и `Notification.requestPermission()` — есть фолбэки, если API не поддержан (`Notification === undefined`).

## Технические детали `localStorage`

Ключи:
- `day-runner-state` — `{ date: "YYYY-MM-DD", items: {itemId: true}, timer: { presetKey, sprints } }`. Главный объект, сбрасывается в новый день.
- `day-runner-sound` — `"on"` | `"off"` (тумблер звука).
- `day-runner-sound-preset` — имя пресета (`bleep` | `chime` | `ding` | `triple` | `low` | `custom`).
- `day-runner-sound-file` — data URL загруженного аудио (если есть).
- `day-runner-sound-file-name` — имя файла для отображения на кнопке.
- `day-runner-resumes` — `{ goal: 10, days: { "YYYY-MM-DD": n } }`. История счётчика резюме, копится по датам, не сбрасывается.
- `day-runner-gh-token` — fine-grained PAT для записи STATS.md (вводит пользователь, секрет).
- `day-runner-gh-synced` — снапшот `days` на момент последнего удачного синка STATS.md.

Дата для ежедневного сброса — `YYYY-MM-DD` локального времени, проверяется на каждом тике.
